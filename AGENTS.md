# hk-config Project Instructions

`hk-config` is the shared hk configuration repo for 2h2d projects. Keep presets composable: project `hk.pkl` files amend hk's `Config.pkl` directly, import `Base.pkl` and stack-specific step maps, and spread them into one hooks map.

## Commit and release conventions

- Use [Conventional Commits](https://www.conventionalcommits.org/) for every commit.
- `Base.pkl` validates commit messages with Cocogitto when `cog` is installed and falls back to hk's conventional-commit utility when it is not.
- Release commits must use exactly:

  ```text
  release: vX.Y.Z
  ```

- Release tags must use the matching `vX.Y.Z` tag.
- `cog.toml` sets `tag_prefix = "v"` and adds `release` as a custom commit type. The `release` type is omitted from generated changelogs so old release commits do not appear as user-facing changes in later releases.
- `CHANGELOG.md` is generated by Cocogitto. Keep Cocogitto's `- - -` separators intact.
- `PklProject` publishes Pkl package artifacts whose package URI has this shape: `package://github.com/2h2d-co/hk-config/releases/download/vX.Y.Z/hk-config@X.Y.Z`.

## Local release flow

1. Make regular changes with conventional commit messages such as `feat: ...`, `fix: ...`, `docs: ...`, or `ci: ...`.
2. Ensure hk hooks are installed, e.g. `hk install --mise` or the global equivalent.
3. Create the release commit and tag with the target version:

   ```sh
   scripts/release.sh X.Y.Z
   ```

   The release script uses `cog changelog` to render the unreleased range, updates and stages `CHANGELOG.md`, creates the signed `release: vX.Y.Z` commit, and creates the matching `vX.Y.Z` tag.

4. Push the release commit and tag:

   ```sh
   git push origin main vX.Y.Z
   ```

## GitHub releases and downstream imports

Pushing a `vX.Y.Z` tag runs `.github/workflows/release.yml`, which packages the Pkl modules, generates GitHub Artifact Attestations for the release assets, and creates the immutable GitHub Release from the matching `CHANGELOG.md` section.

Downstream repos should amend the matching hk `Config.pkl` and pin imports to release packages, for example:

```pkl
amends "package://github.com/jdx/hk/releases/download/v1.50.0/hk@1.50.0#/Config.pkl"
import "package://github.com/2h2d-co/hk-config/releases/download/vX.Y.Z/hk-config@X.Y.Z#/Base.pkl" as Base
```

Use the same package version for every imported library module.

## Validation

Before handing off changes, run:

```sh
hk validate
hk check --all --check
tmp=$(mktemp -d)
HK_CONFIG_VERSION=0.0.0 pkl project package --skip-publish-check --output-path "$tmp" .
rm -rf "$tmp"
```

Library modules are validated with `pkl eval`; only the project `hk.pkl` is an hk configuration module. For GitHub Actions changes, also evaluate the library directly:

```sh
pkl eval GitHubActions.pkl >/dev/null
```
